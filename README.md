# CCTV 이력 관리 시스템

CCTV 영상을 VLM이 보고 "누가 무슨 행동을 했는지"를 5초 구간마다 기록해 두고, 자연어로 검색하는 관제 콘솔.

> 예) "쓰러진 사람 찾아줘" → 그 장면·카메라·시각을 바로 표시
> 로컬 PC(RTX 4060 8GB)에서 동작. 화재는 열화상 카메라가 따로 담당.

<br>

## 개발 목적

- 현대글로비스 선박의 미확인 화재사고를 방지하기 위해 CCTV(일반·열화상) 실시간 탐지 알림 시스템 개발 수주
- → 실시간 알림을 기점으로, 알림이 있기 전에 해당 구역에 어떤 상황들이 있었고 화재로 이어졌는지 원인을 분석할 수 있도록 LLM이 미리 이력을 저장해두는 시스템

<br>

## 개발 도구

| 구분 | 사용 |
|---|---|
| 기술 스택 | Python · FastAPI · React |
| VLM 모델 | InternVL3-8B / Qwen3-VL-8B / Qwen2.5-VL-7B |
| 임베딩 | bge-m3 (다국어) |
| 벡터 DB | ChromaDB (HNSW, cosine) |

<br>

## 사용 데이터

검증·PoC 데이터로 AI-Hub "07.지능형 관제 서비스 CCTV 영상 데이터" 활용.

- 6종 행동 라벨: 쓰러짐 · 싸움 · 침입 · 군집 · 인파밀집 · 침수
- 각 영상에 정답 라벨 포함: 유형 + 사건 발생 프레임 + 장면 설명
- 라벨은 AI에 안 보여주고, AI가 영상만 보고 맞히게 한 뒤 **채점(정답지)** 으로만 사용

```json
{
  "metadata": {
    "file_name": "E06_001.mp4",
    "width": 1920, "height": 1080,
    "frame_count": 8967,
    "date": "241018",                 // 녹화일 (2024-10-18)
    "night": false
  },
  "annotations": {
    "event_class": "Flood",           // 유형 정답
    "event_frame": [[2, 8967]],       // 사건 발생 프레임 [시작, 끝]
    "event_length": 298.83,           // 사건 길이(초)
    "event_caption": "비가 내려 웅덩이가 생긴 침수된 도로에 다수 자동차가 지나감"
  }
}
```

## 워크플로우

```
CCTV 영상
  ├─ ① 5초 구간으로 분할 + 프레임 추출 (OpenCV·Pillow)
  ├─ ② VLM이 구간마다 장면 묘사
  ├─ ③ 묘사를 임베딩해 벡터 DB 저장 (구간 1개 = 1줄)
  ├─ ④ 묘사 ↔ 유형 설명 유사도로 유형 분류 (재색인 없이 교체)
  └─ ⑤ 자연어로 검색 (RAG)
```

<br>

### 1. CCTV 영상 입력 (AI-Hub)

- AI-Hub "지능형 관제 CCTV" 영상을 입력으로 사용
- 6종 행동 라벨을 채점의 정답지로 사용 (AI에는 비공개)

<br>

### 2. 5초 구간 분할 + VLM 입력 프레임 추출 (OpenCV · Pillow)

영상을 5초 구간으로 자르고, 각 구간에서 대표 프레임 N장(2/6/16)을 균일 간격으로 뽑아 VLM에 시계열로 투입.

**프레임 추출 (OpenCV)**

- `cv2.VideoCapture` 로 영상을 열고 `FPS` · `FRAME_COUNT` 로 전체 길이 계산
- 5초 그리드로 빠짐없이 분할
- 구간 내 N프레임을 균일 간격으로 배치
  - `idx = (start + (end-start)·(i+1)/(N+1)) · fps`
- OpenCV는 BGR, VLM/PIL은 RGB → `cvtColor(BGR2RGB)` 로 색공간 변환
- `cap.set(POS_FRAMES, idx)` + `read()` 로 필요한 프레임만 seek (전체 디코드 안 함)

**프레임 전처리 (Pillow)**

- `Image.fromarray()` 로 OpenCV(numpy) 프레임을 PIL 이미지로 변환 (VLM 프로세서 입력 형식)
- 긴 변이 560px 초과 시 비율 유지 축소 (`resize`, BILINEAR) → OOM을 '프레임 수'가 아니라 '해상도'로 관리
- 구간 중앙 프레임 1장은 썸네일로 저장 (`save`, quality=85) → 콘솔 표시·RAG 근거 프레임

**[실험 A] 프레임을 늘려도 정확도가 올라가지 않음**

| 모델 | 2프레임 | 6프레임 | 16프레임 |
|---|---|---|---|
| InternVL3-8B | 42% | 42% | 42% |
| Qwen3-VL-8B | 21% | 26% | 16% |
| Qwen2.5-VL-7B | 47% | 37% | 37% |

**[실험 B] 입력 방식(이미지 vs 비디오)에 따라 모델별 정확도가 갈림**

- 추출 프레임을 `{"type":"video"}` 로 한 번에 투입 (`apply_chat_template` + `processor(videos=)`)
- 토큰 경제가 이유: 같은 프레임이라도 입력 방식에 따라 토큰량이 3배 차이

| 입력 방식 | 16프레임 토큰 | 비고 |
|---|---|---|
| N개 이미지 | 8장 ≈ 12.5k tok | 프레임마다 타일링 → 토큰 폭증 |
| video 묶음 | 16장 ≈ 4.4k tok | 시간 병합 + 프레임당 고정 격자 → 1/3 |

- 덕분에 로컬 8GB에서도 멀티프레임 가능 (video 미지원 모델은 이미지 경로로 자동 폴백)

| 모델 | 2f 이미지 | 2f video |
|---|---|---|
| InternVL3-8B | 37% | 42% |
| Qwen3-VL-8B | 53% | 21% |
| Qwen2.5-VL-7B | 11% | 47% |

→ Qwen3-VL은 이미지를, 나머지는 비디오를 더 잘 인식.

**프레임별 VRAM·속도** (InternVL3-8B 4bit @ 8GB)

| 프레임 | VRAM | 추론 | 비고 |
|---|---|---|---|
| 4장 | 6.1GB | 2초 | |
| 6장 | 6.5GB | 2초 | 로컬 기본(스필 없는 한계) |
| 8장 | 7.1GB | 15초 | 스필 시작 |
| 16장 | 10.9GB | 49초 | 서버 전용 |

**[실험 C] 객체 추적 기반 프레임 추출 (시도 후 폐기)**

- 목적: CCTV 속에 작게 찍힌 사람의 행동을 VLM이 놓치지 않도록 추적으로 보강
- YOLOv8n(탐지): 매 프레임에서 사람·차량 위치를 박스로 찾음 (누가 누군지는 모름)
- ByteTrack(추적): 프레임 간 박스를 이어 같은 객체를 특정 → 객체별 위치·크기 변화 산출

| 모델 | 객체추적 X | 객체추적 O |
|---|---|---|
| InternVL3-8B | 42% | 32% |
| Qwen3-VL-8B | 16% | 11% |
| Qwen2.5-VL-7B | 37% | 32% |

→ 오히려 정확도가 하락하여 균등 프레임 방식 채택. 추적 코드(YOLOv8n·ByteTrack)는 제거.

### 3. VLM 영상 묘사

입력 프롬프트와 5초 구간 프레임(비디오)을 기반으로 상황 묘사 텍스트를 출력.

```python
SEGMENT_EVENT_PROMPT = (
    "다음 연속 프레임은 같은 5초 구간을 시간순으로 본 것입니다. "
    "이 구간에서 사람과 사물이 실제로 무엇을 하는지 객관적 사실로 한국어로 묘사하세요.\n"
    "기호 없이 순수한 텍스트로 아래 한 줄 형식만 작성하세요.\n\n"
    "사람의 구체적 행동(밀치거나 다투기, 넘어져 쓰러지기, 담을 넘거나 한곳에 오래 머무르기, "
    "좁은 곳에 밀집하기)이나 바닥 침수가 보이면 그것을 캡션 맨 앞에 먼저 적으세요. "
    "보이지 않는 상황을 '없음'이라고 나열하지 말고, 화면에 실제로 보이는 것만 묘사하세요.\n\n"
    "캡션: 장면을 1~2문장으로 구체적으로"
)
```

- 입력: 5초 구간 프레임(비디오)
- 출력: `사람이 도로에서 넘어져 쓰러지고 있습니다.`

<br>

### 4. VLM 묘사 임베딩 → 벡터 DB 저장

- 출력 묘사를 임베딩 모델(bge-m3)로 벡터화해 ChromaDB에 저장
- 5초 구간 정보(시각·영상·썸네일)를 메타데이터로 함께 저장
- 구간 1개 = 레코드 1줄 (묘사 원문 그대로, 요약·병합 안 함 → 검색 시 의미 희석 없음)

```python
{
  "id": "fall_E02_041:s12",
  "document": "사람이 도로에서 넘어져 쓰러지고 있습니다.",   # ← 이게 임베딩됨(벡터화)
  "metadata": {
    "start_s": 60, "end_s": 65,                          # ← 시각
    "video_id": "fall_E02_041",
    "event_type": "unknown",                             # ← 유형은 5단계에서 별도로 부착
    "thumb": "...", "indexed_at": "..."
  }
}
# store.add → col.upsert(ids=, documents=, metadatas=)
```

### 5. 영상 유형 분류 (오픈셋)

영상의 유형을 6가지로 나누어 메타데이터에 추가: 쓰러짐 · 싸움 · 침입 · 군집 · 인파밀집 · 침수.

두 방식을 비교:

- **(a) 직접 분류**: VLM이 클립을 보고 유형을 직접 분류
- **(b) 임베딩 유사도**: VLM은 묘사만 하고, '묘사 텍스트 ↔ 클래스 설명' 벡터 유사도로 유형 별도 분류
  - 클래스 설명 = 각 유형의 설명으로 구성된 json 파일(event_classes.json)

```json
[
  { "key": "falldown",  "label": "쓰러짐",   "desc": "사람이 갑자기 넘어지거나 계단에서 굴러 바닥에 쓰러져 누움" },
  { "key": "fight",     "label": "싸움",     "desc": "한 사람이 다른 사람의 멱살·머리채를 잡고 밀치며 몸싸움·폭행이 발생" },
  { "key": "invasion",  "label": "침입",     "desc": "사람이 펜스·담장을 넘어 학교나 제한구역으로 무단 침입" },
  { "key": "gathering", "label": "군집",     "desc": "걷던 사람들이 멈춰 서서 한곳에 모여 군집을 형성하고 대화함" },
  { "key": "crowd",     "label": "인파밀집", "desc": "다수의 사람이 골목·상가·거리에 빽빽하게 몰려 인파밀집을 이룸" },
  { "key": "flood",     "label": "침수",     "desc": "비로 도로·골목이 침수되어 물웅덩이가 형성되고 물에 잠김" }
]
```

**결과**

| 방식 | 정확도 |
|---|---|
| (a) VLM 직접 분류 | 26% |
| (b) 임베딩 유사도 | 37% |

→ **(b) 채택**. 유형 목록(json)만 바꾸면 재색인 없이 즉시 재분류 가능하고, VLM 사용량을 줄여 빠르게 분류.

### 6. CCTV 이력 페이지에서 자연어 검색 (RAG)

- **입력**: "쓰러진 사람 찾아줘"
- **처리 흐름**
  - 질문을 묘사와 같은 임베딩 모델(bge-m3)로 벡터화
  - 저장된 5초 구간 캡션과 벡터 유사도 비교 (유사도 임계 0.52 미만은 '무관'으로 제외)
  - 같은 영상에서 20초 이내 구간은 '같은 상황'으로 보고 최고점 1개만

| 단계 | 단위 | 이유 |
|---|---|---|
| 검색(매칭) | 5초 구간 | 구간 캡션이 깔끔한 검색 단위 → 정확히 매칭 (사건으로 요약·병합하면 의미 희석됨) |
| 중복 제거 | 시간 근접(20초) | 같은 상황이 인접 구간에 반복 매칭돼도 대표 1개만 표출 |

- **출력 결과**
  - 카메라 번호 · 시각 · 상황묘사 · 유사도 점수 · 썸네일
  - 의미 매칭: "자전거" → "자전거가 바닥에 넘어져 있고, 한 사람이 자전거를 고치고 있습니다." (단어가 달라도 뜻이 통하면 검색됨)
  - 결과를 클릭하면 해당 카메라·시각으로 영상이 점프

<br>

## 핵심 설계

- **묘사와 유형 분류를 분리**: VLM은 묘사만, 유형은 임베딩 유사도로 따로. 유형 목록을 바꿔도 영상 재분석(수십 분) 없이 몇 초 만에 재분류.
- **구간 단위 저장·검색**: 사건으로 묶지 않고 5초 구간을 그대로 저장. 자동 사건분리가 부정확해 구간 직접 검색 + 시간근접 중복 제거가 더 견고.
- **실제 시각 기준**: 녹화 시작시각 + 구간 오프셋으로 관제용 절대 시각 표기.
