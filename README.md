# cctv_memory

CCTV 녹화 영상에서 VLM이 사람 행동을 **사건(event)** 으로 뽑아 색인하고,
자연어로 검색하며 카메라·시각별 이력으로 관리하는 관제 시스템.

> 로컬 RTX 4060 8GB (4bit). 입력은 저장된 CCTV 파일.
> 화재는 열화상 카메라 담당, 본 시스템은 행동 이력에 집중.

이 문서는 실행법이 아니라 **설계가 어떻게, 왜 바뀌었는지**를 정리한다.

<br>

## 한눈에: 설계 변천

| 영역 | 이전 | 이후 | 왜 바꿨나 |
|---|---|---|---|
| 목적 | 화재 위험 분류 | 사람 행동 이력 | 화재는 열화상이 더 잘함. VLM은 행동 해석에 집중 |
| 저장 단위 | 5초 구간 | 사건 (구간 병합) | "이력"은 사건 단위라야 읽힘 |
| 시간·체류 | VLM이 판단 | ByteTrack 추적 | VLM은 체류시간을 못 봄. 배회는 dwell로 결정 |
| 유형 분류 | VLM 고정 taxonomy (A) | 캡션 임베딩 오픈셋 (B) | 유형 바꿔도 재색인 불필요 |
| 캡션 프롬프트 | 단서 유도형 | 중립 묘사형 | (B)에선 캡션이 분류 재료라 깨끗해야 |
| taxonomy | 임의 6종 | AI-Hub GT 6종 | 공정한 평가 기준 |
| 시각 | 영상 오프셋 (00:15) | 절대시각 (10:22) | 관제는 절대시각이 기본 |
| VLM 선정 | 이미지 분류로 평가 | 영상 색인으로 벤치 | 실제 쓸 작업 기준으로 |

<br>

## 핵심 1. 색인과 분류를 분리 (A → B)

**(A) 옛 방식**
- 색인할 때 VLM이 캡션 + 유형을 같이 결정
- 문제: 새 유형(싸움·침입) 추가하려면 **전체 재색인** (VLM, 수십 분)

**(B) 현재**
- 색인: VLM은 **캡션(자유 묘사)만**
- 분류: 캡션 임베딩 ↔ 클래스 설명 임베딩 유사도 (`classifier.py` + `event_classes.json`)
- 유형 변경: `reclassify.py`만 실행 (몇 초, **VLM 안 돎**)

**효과**
- 클래스 무한 교체 (현장마다 다른 유형)
- VLM 출력이 짧아져 색인도 빨라짐
- Twelve Labs의 index ↔ classify 분리와 같은 발상

<br>

## 핵심 2. 캡션 프롬프트 중립화

- 옛 프롬프트: "흡연·화기·쏟아짐" 같은 단서를 박음
- 부작용: 인파밀집 장면을 *"액체가 쏟아져 있습니다"* 로 오염 (단서 누수)
- 바꿈: **"분류 말고, 무슨 일이 일어나는지 사실대로 묘사"**
- 이유: (B)는 캡션이 분류 재료. 깨끗(중립)해야 어떤 클래스에도 맞음

<br>

## 평가: A vs B vs GT

- GT = AI-Hub 71850 **실제 라벨** (정답 유형 + 정확한 시각 + 캡션)
- `eval_aihub.py` 가 셋을 비교:

| 방식 | 무엇 | 한계 |
|---|---|---|
| A | VLM 직접 분류 | 고정 taxonomy라 싸움·침입·군집을 못 잡음 |
| B | 캡션 임베딩 분류 | 클래스만 있으면 잡음 (오픈셋) |
| GT | 실제 라벨 | 정답 |

<br>

## 현재 아키텍처

```
영상
 → tracker (YOLO+ByteTrack)   사람·차량 ID·체류시간
 → segmenter (5초 구간)
 → VLM (중립 캡션)
 → event_builder (사건 병합)
 → ChromaDB
 → [조회 시] classifier (오픈셋 유형)
 → 콘솔 (검색·이력·알림)
```

역할 분담
- 사실 (체류·시각·입출입): 추적 + `cctv_meta`
- 의미 (행동): VLM
- 유형: 임베딩 분류 (교체 자유)

<br>

## 실 운영 수집 (설계 방향)

- NVR / ffmpeg 가 RTSP를 시간분할 파일로 저장 (녹화는 GPU 무관)
- 폴더 워처가 **완결된 파일만** 큐에 적재
- 단일 GPU 워커가 배치 색인
- 절대시각은 파일명에서 복원: `CAM03_20260611_093000.mp4`
